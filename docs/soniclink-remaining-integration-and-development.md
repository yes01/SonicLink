# SonicLink 剩余联调与开发任务

## 文档目的

本文档整理 `soniclink-development-checklist.md` 中尚未完成的内容，并按“继续开发项”和“联调/验证项”拆分。当前 APK、平台后端、平台前端已经具备基础 Agent 注册、心跳、远控命令、H.264 投屏转发和设备展示能力；剩余内容主要集中在增强能力、稳定性、真实设备验证和产品化决策。

## 一、继续开发项

### 1. 环境与构建能力

- build flavor：
  - 内容：如需要区分测试环境和正式环境，增加 `debug/staging/release` 或类似 flavor。
  - 涉及仓库：`SonicLink`
  - 建议优先级：P2
  - 验收标准：不同 flavor 可配置不同默认服务器地址、应用名称或版本后缀；GitHub Actions 可按 flavor 构建 APK。

- wake lock：
  - 内容：长时间运行或长时间投屏时按需增加 wake lock，避免设备休眠导致 Agent 断流。
  - 涉及仓库：`SonicLink`
  - 建议优先级：P1
  - 验收标准：连续投屏 30 分钟期间，设备不因休眠导致 WebSocket 或 MediaProjection 中断；停止投屏后能释放 wake lock。

### 2. 网络与安全增强

- TLS/WSS：
  - 内容：跨网使用前补充 HTTPS/WSS 支持。
  - 涉及仓库：`SonicLink`、`xin_testing_platform`
  - 建议优先级：P2
  - 验收标准：APK 可配置 `wss://` 平台地址；平台后端反向代理或服务端证书配置可用；token 不通过明文公网传输。

- Google Play 无障碍政策评估：
  - 内容：若未来通过 Google Play 分发，需要评估 Accessibility API 使用说明、权限披露和审核风险。
  - 涉及仓库：产品/发布文档
  - 建议优先级：P3
  - 验收标准：形成发布策略，明确是否走企业分发、内部分发、侧载或 Play Store。

### 3. 投屏质量与资源控制

- 自适应码率：
  - 内容：根据网络情况、设备性能或前端反馈动态调整码率、帧率、分辨率。
  - 涉及仓库：`SonicLink`、`xin_testing_platform`
  - 建议优先级：P1
  - 验收标准：弱网或多设备场景下自动降码率；恢复后可提升质量；前端延迟和卡顿明显下降。

- 丢帧数统计：
  - 内容：补充视频发送、转发、前端解码链路中的丢帧统计。
  - 涉及仓库：`SonicLink`、`xin_testing_platform`
  - 建议优先级：P1
  - 验收标准：平台设备详情中可看到 Agent 上报帧数、平台转发帧数、前端接收帧数，能推算丢帧。

- 带宽估算：
  - 内容：基于视频字节数和时间窗口估算单设备和总带宽。
  - 涉及仓库：`xin_testing_platform`
  - 建议优先级：P1
  - 验收标准：平台展示每台设备近实时 Mbps，总带宽可聚合。

- Mac CPU 负载估算：
  - 内容：平台端展示 Mac 当前 CPU 负载，用于判断多设备投屏容量。
  - 涉及仓库：`xin_testing_platform`
  - 建议优先级：P2
  - 验收标准：设备远控页或监控接口可显示 CPU 负载，并能与投屏数量关联观察。

- 每台设备独立投屏质量配置：
  - 内容：允许按设备设置分辨率、码率、帧率、I 帧间隔。
  - 涉及仓库：`SonicLink`、`xin_testing_platform`
  - 建议优先级：P1
  - 验收标准：前端可修改单台设备投屏质量；后端下发 `start_stream` payload；APK 按配置启动编码。

### 4. 输入能力补充

- 自定义输入法兜底：
  - 内容：保留或改造现有 Sonic 自定义输入法，作为无障碍文本输入失败时的兜底。
  - 涉及仓库：`SonicLink`
  - 建议优先级：P2
  - 验收标准：当前焦点不可通过无障碍 `ACTION_SET_TEXT` 输入时，可切换到输入法方案完成文本输入。

- 剪贴板粘贴兜底：
  - 内容：评估剪贴板方式作为文本输入补充方案。
  - 涉及仓库：`SonicLink`
  - 建议优先级：P3
  - 验收标准：只有在用户明确允许或调试模式下使用；不默认记录或泄露敏感文本。

### 5. 音频能力

- 独立音频流：
  - 内容：如果后续需要远程听音，基于现有音频服务补充独立音频流。
  - 涉及仓库：`SonicLink`、`xin_testing_platform`
  - 建议优先级：P3
  - 验收标准：音频采集、编码、传输和前端播放可开关控制；不影响首版屏幕投流稳定性。

### 6. 稳定性边界处理

- 无障碍服务运行中被关闭：
  - 内容：Agent 心跳和 UI 状态实时感知无障碍被关闭，并阻断远控命令。
  - 涉及仓库：`SonicLink`、`xin_testing_platform`
  - 建议优先级：P1
  - 验收标准：用户关闭无障碍后，平台设备状态更新；点击、滑动、文本输入返回明确错误；APK 页面提示重新开启。

- 息屏和锁屏：
  - 内容：处理设备息屏、锁屏后投屏黑屏、控制失败、唤醒等场景。
  - 涉及仓库：`SonicLink`
  - 建议优先级：P1
  - 验收标准：息屏后状态明确；支持唤醒或提示用户无法控制；锁屏场景不出现无反馈失败。

### 7. 自动化测试补充

- instrumentation 手势测试：
  - 内容：为手势构造和无障碍控制增加可在真机或模拟器上运行的 instrumentation 测试。
  - 涉及仓库：`SonicLink`
  - 建议优先级：P2
  - 验收标准：CI 或本地可运行基础点击、滑动、长按测试；失败时能定位是权限、服务状态还是坐标问题。

### 2. 平台后端依赖与单测验证

- 当前阻塞：
  - 当前全局 Python 环境没有 `pytest`，无法执行平台单测。
- 联调内容：
  - 安装后端依赖。
  - 执行 Agent 管理器单测。
  - 执行数据库迁移。
- 建议命令：
  - `cd /Users/lx/Downloads/workspace/xin_testing_platform/backend`
  - `pip install -r requirements-common.txt`
  - `python3 -m pytest tests/test_soniclink_agent_manager.py`
  - `python3 cli.py upgrade`
- 验收标准：
  - pytest 通过。
  - Alembic 迁移成功，`device_remote_devices` 表包含 Agent 字段。

### 3. APK 与平台基础联调

- 联调内容：
  - GitHub Actions 构建 APK 或使用本地 debug APK。
  - 手机安装 APK。
  - 配置平台 WebSocket 地址。
  - 开启无障碍。
  - 授权屏幕录制。
  - 点击启动 Agent。
- APK WebSocket 地址格式：
  - `ws://<Mac局域网IP>:<后端端口>/v1/device_remote/agent/ws`
- 如果启用平台 token：
  - 后端设置 `SONICLINK_AGENT_TOKEN`。
  - APK 设置页填写同一个 token。
- 验收标准：
  - 平台后端收到 Agent 注册。
  - 远程设备列表出现 `SonicLink Agent` 设备。
  - 心跳持续更新。
  - 平台能展示无障碍、投屏授权、电量、网络等状态。

### 4. 控制联调

- 联调内容：
  - 点击桌面 App 图标。
  - 滑动桌面。
  - 长按。
  - 返回、Home、最近任务。
  - 文本输入。
  - 双指缩放。
- 验收标准：
  - 坐标映射准确。
  - 命令结果有成功/失败回调。
  - 无障碍关闭时错误提示明确。
  - 双指缩放在支持页面内生效。

### 5. 投屏联调

- 联调内容：
  - 反复开始和停止投屏。
  - 投屏中旋转屏幕。
  - 连续运行 30 分钟。
  - 两台以上设备同时投屏。
- 验收标准：
  - 前端画面可播放。
  - 旋转后画面尺寸和坐标映射正确。
  - 连续运行 30 分钟无明显内存增长、黑屏、断流。
  - 多设备同时投屏时互不串流。

### 6. 多设备容量联调

- 联调内容：
  - 目标局域网内测试 2 台、4 台、8 台设备同时在线。
  - 观察 Mac CPU、内存、带宽、前端播放延迟。
  - 验证 `SONICLINK_MAX_AGENT_STREAMS` 同时投屏上限。
- 验收标准：
  - 多设备列表稳定在线。
  - 超过上限时返回明确错误。
  - 已在线设备断开后，服务端清理在线状态和流会话。

### 8. 延迟目标验证

- 验证项：
  - 局域网内手势命令往返低于 200 ms。
  - 首版视频端到端延迟低于 500 ms。
  - WiFi 或服务中断后 10 秒内自动重连。
- 建议方式：
  - 后端记录命令下发和响应时间。
  - 前端记录视频帧接收时间。
  - 使用秒表页面或屏幕闪烁页面辅助观察端到端延迟。
- 验收标准：
  - 达到上述目标，或记录不达标场景并进入自适应码率/网络优化任务。

## 三、产品与架构决策项

- 包名是否保留 `org.cloud.sonic.android`：
  - 当前状态：仍保留原包名。
  - 决策建议：如果未来作为独立产品发布，建议改成独立包名；如果仅内部测试使用，可暂时保留。

- 现有 TCP socketmanager 模块是否保留：
  - 当前状态：legacy 服务保留为非导出。
  - 决策建议：短期保留，避免影响旧能力；Agent 通道稳定后再逐步下线。

## 四、建议推进顺序

1. 先完成平台前端依赖安装和构建验证。
2. 完成平台后端依赖安装、pytest 和数据库迁移验证。
3. 用 1 台 Android 设备完成 APK 到平台的端到端联调。
4. 补齐无障碍关闭、息屏/锁屏这两个 P1 稳定性场景。
5. 做 2 台设备同时投屏联调，再扩展到 4 台和 8 台。
6. 根据多设备测试结果开发自适应码率、带宽估算和每设备质量配置。
7. 最后处理产品化事项：TLS/WSS、发布策略、包名、legacy 模块下线计划。
